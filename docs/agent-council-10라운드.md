## 에이전트별 핵심 주장

### 1. 헥사고날 아키텍처 전문가 (Architect) — "적극 권장"

**입장**: 적용 가능하며, 지금이 최적의 시점

- 현재 `course-data.ts`가 God Object (데이터 + 로직 + 리포지토리 혼재)
- 45개 대학 확장 시 이 파일이 수천 줄로 팽창 예측
- Next.js SSG와 헥사고날은 **충돌하지 않음** — 빌드타임이 Driving Adapter 역할
- `generateStaticParams()`가 인바운드 어댑터, JSON 파일 읽기가 아웃바운드 어댑터

**제안 구조**:
```
domain/ (순수 TS, 외부 의존 없음)
├── models/      → University, Course, Subject
├── ports/       → inbound (유스케이스), outbound (리포지토리 인터페이스)
└── services/    → GetCourseRecommendation, CompareUniversities

infrastructure/
├── adapters/    → JsonFileRepository (현재), ApiRepository (향후)
└── di/          → container.ts (어댑터 선택/주입)

app/             → Next.js Pages (인바운드 어댑터)
components/      → Client Components (UI만)
```

**권고**: 옵션 C (점진적 헥사고날) — Port 인터페이스 먼저, 기존 하드코딩을 InMemoryAdapter로 래핑

### 2. Next.js SSG 프론트엔드 전문가 (Planner) — "라이트 헥사고날"

**입장**: 풀 헥사고날은 과잉, 원칙만 취하라

- 846줄에 풀 헥사고날은 **과잉(over-engineering)**
- **"헥사고날 아키텍처를 도입한다"보다 "관심사를 올바른 방향으로 분리한다"가 맞는 프레임**
- 형식적 인터페이스는 실제 필요가 발생할 때 추출

**점진적 3단계 전략**:

| 단계 | 시기 | 내용 | 비용 |
|------|------|------|------|
| Phase 1 | 즉시 | 데이터 레이어 분리 (`data-loader.ts` + `course-utils.ts`) | 1시간 |
| Phase 2 | 45개 대학 확장 시 | `generateStaticParams()` + 동적 라우트 | 3-4시간 |
| Phase 3 | 데이터 소스 다변화 시 | 정식 인터페이스 + 어댑터 패턴 | 필요 시 |

**핵심 메시지**:
> "인터페이스와 어댑터라는 형식은 실제 필요가 발생할 때 자연스럽게 추출하면 됩니다."

### 3. 실용주의 비평가 (Critic) — "하지 마십시오"

**입장**: YAGNI의 교과서적 사례. 분석 마비 재발.

**핵심 논거**:

| 기준 | 판정 |
|------|------|
| YAGNI | **위반** — DB 없음, API 없음, 비즈니스 로직 40줄 |
| ROI | **역전** — 같은 17-24시간에 대학 15개 추가가 10배 가치 |
| 과도 설계 | **명백** — 842줄 + 수명 2년 = 엔터프라이즈 아키텍처 부적합 |
| 분석 마비 | **재발** — 9라운드 분석 마비의 변형 |

**시간 비용 비교**:
- 헥사고날 도입: **17-24시간** (2-3일)
- 같은 시간의 대안:
  - 대학 15개 추가 (12-16시간) → UV 직접 증가
  - `data/universities/` JSON 연동 (4-6시간) → 기능 완성도
  - SEO 최적화 (4-6시간) → 검색 유입
  - 역방향 검색 구현 (6-8시간) → 핵심 차별화

**결론**:
> "완벽한 아키텍처를 가진 죽은 프로젝트보다 지저분하지만 사용자가 있는 살아 있는 프로젝트가 낫습니다."
> "3개월 후 UV 2,000 미달 시 피봇한다면서, 2-3일을 아키텍처 리팩토링에 쓸 겁니까?"

### 4. DDD/클린 아키텍처 전문가 (Analyst) — "부분 적용 + 숨은 버그 발견"

**입장**: 풀 DDD/헥사고날은 불필요하나, 도메인 모델링으로 현재 버그를 해결할 수 있다

**발견한 숨은 문제들**:

#### 버그 1: 과목명 정규화 불일치
```
서울대: "확률과 통계" (띄어쓰기)
연세대~중앙대: "확률과통계" (붙여쓰기)
→ compareUniversities()에서 같은 과목이 다른 과목으로 인식됨
```

#### 버그 2: "또는" 관계 미모델링
```typescript
"지구환경과학부": { core: "물리학Ⅱ 또는 화학Ⅱ" }
→ split(",") 후 단일 항목으로 처리
→ 역방향 검색 시 "물리학Ⅱ"로 이 학과를 찾을 수 없음
```

#### 버그 3: 학과명 매칭 오류
```typescript
depts.find(d => d.includes(department) || department.includes(...))
→ "공학"으로 검색 시 모든 공학과 매칭 (오탐)
→ Array.find()는 첫 번째 매칭만 반환 (누락)
```

#### 버그 4: 데이터 소스 단절
```
course-data.ts: 6개 대학 (UI에서 사용)
data/universities/*.json: 15개 대학 (미연동, 방치)
→ 정합성 보장 없음
```

**적용해야 하는 것** (High Value, Low Cost):

| 항목 | 이유 | 비용 |
|------|------|------|
| Subject Value Object | 과목명 정규화 버그 해결 | 1개 파일 |
| Repository Interface | JSON→API 전환 대비 | 1개 인터페이스 |
| 데이터/로직 분리 | God Object 해체 | 3-4개 파일 |
| 유스케이스 함수 분리 | 역방향 검색 추가 용이 | 함수 이동 |

**적용하지 말아야 하는 것**:
- 정식 Aggregate Root 클래스 (읽기 전용이므로)
- Domain Events (쓰기 작업 없음)
- CQRS (단일 읽기 모델로 충분)
- Application Service 클래스 (함수로 충분)
- Anti-Corruption Layer (외부 시스템 없음)

## 최종 권고: "실용적 관심사 분리"

### 하지 말 것
- 정식 Hexagonal Architecture (포트 인터페이스, DI 컨테이너, 어댑터 패턴)
- DDD 전체 도입 (Aggregate Root, Domain Events, CQRS)
- 유스케이스 클래스 (함수로 충분)

### 할 것 (Phase 1과 동시에)

**1단계: 데이터/로직 분리 (1-2시간)**
```
web/src/lib/
├── types.ts           # 타입 정의 (현행 유지)
├── data-loader.ts     # JSON 파일 로드 함수 (신규, ~30줄)
├── course-utils.ts    # 순수 도메인 함수들 (현행 로직 이동)
└── subject.ts         # Subject 정규화 + 카테고리 (버그 수정)
```

**2단계: 숨은 버그 수정 (1시간)**
- 과목명 정규화 ("확률과 통계" = "확률과통계")
- "또는" 관계 파싱 ("물리학Ⅱ 또는 화학Ⅱ" → 2개 옵션)
- 학과명 매칭 개선 (부분 문자열 → 키워드 기반)

**3단계: 필요할 때만 (UV 2,000+ 달성 후)**
- Repository 인터페이스 추출
- 어댑터 패턴 도입
- 정식 포트/어댑터 분리
