# Agent Council 7라운드: "두 명의 개발자 + Claude Code = 게임 체인저?"

> 분석일: 2026-03-01
> 참여 에이전트: Claude Code 파워유저, 스타트업 속도 전문가, 현실주의 비판자, 2인 팀 전략가
> 주제: 2인 개발자 + Claude Code 전면 활용 시 프로젝트 성공 가능성 재평가

---

## 핵심 결론 (스포일러)

> **4명 중 2명은 "속도는 근본 문제를 해결하지 못한다"고 반대했고, 2명은 "2인+Claude Code+국책사업 조합이면 기대값이 뒤집힌다"고 찬성했습니다.**
>
> **결론: "두 명이면 빠르다"는 맞지만, "빠르면 성공한다"는 틀립니다. 단, 국책사업이라는 변수가 기대값을 -68.5만원에서 +500~1,800만원으로 뒤집을 수 있습니다.**

## 1. Claude Code 파워유저: "코딩은 3-5배 빨라지지만, 데이터는 그대로"

### Claude Code 실제 생산성 배수

| 작업 유형 | 생산성 배수 | 근거 |
|-----------|-----------|------|
| 보일러플레이트/설정 (SSG, sitemap) | **5-8x** | 패턴 명확, Claude Code 즉시 생성 |
| UI 컴포넌트 (Tailwind + React) | **3-5x** | 기존 코드 스타일 반복 |
| 비즈니스 로직 (역인덱스, 매핑) | **3-4x** | 알고리즘 단순 |
| JSON 구조화 (추출된 데이터) | **5-10x** | Claude Code 최강 영역 |
| **데이터 수집 (PDF 파싱)** | **1.0-1.2x** | **Claude Code 도움 불가** |
| **데이터 사실 검증** | **1.0x** | **사람만 가능** |

**종합: 순수 코딩 3-5x, 데이터 포함 시 1.5-2x**

| 데이터 정규화 | 6h | **3.5-4h** | 35-45% |

| OG 이미지 + sitemap | 4h | **0.5-1h** | 75-85% |

| 테스트 작성 | 6h | **1-1.5h** | 75-80% |

### 2인 병렬 시 현실적 타임라인: 4주

| 주차 | 개발자 A (프론트엔드) | 개발자 B (데이터) |
|------|---------------------|------------------|

## 2. 스타트업 속도 전문가: "속도는 실패 원인의 10%만 해결한다"

### 이 프로젝트의 실패 리스크 분해

| 실패 원인 | 비중 | 속도로 해결? |
|-----------|------|------------|
| 수요 부재 (사용자 검증 0회) | **40%** | 불가능 |
| 데이터 품질 (잘린 데이터, 표기 불일치) | **25%** | 빠르면 오히려 악화 |
| 경쟁 (adiga.kr, 진학사 등 기존 서비스) | **15%** | 불가능 |
| 기술 리스크 (SSG 전환, 역검색) | **10%** | **해소 가능** |
| 규제/법적 리스크 (데이터 출처) | **10%** | 불가능 |

**속도로 해소 가능한 리스크는 전체의 10%**

### 2인 팀의 기대값: 오히려 악화?

| 항목 | 1인 67시간 (기존) | 2인+Claude Code |
|------|-------------------|-----------------|
| 총 투입 시간 | 67시간 | 45-50시간 (2인 합계) |
| 시간당 기회비용 | 3만원 x 1인 | 3만원 x 2인 = 6만원 |
| 총 기회비용 | 201만원 | 270-300만원 |
| 성공 확률 | 20-30% | 25-35% (+5%p) |
| **기대값** | **-68.5만원** | **약 -150만원** |

**비용은 1.5배 증가, 성공 확률은 5%p만 상승 → 기대값 악화**

### 대안 프로젝트 기대값 비교 (동일 2인+Claude Code 리소스)

| 프로젝트 | 기대값 | 이유 |
|---------|--------|------|
| **개발자 도구 비교 (B-1)** | **+250~350만원** | 코드 중심 → Claude Code 효과 극대화, 데이터 병목 없음 |
| 하이브리드 (C-1) | +120~180만원 | 리스크 분산 |
| **현재 프로젝트** | **-150만원** | 데이터 병목으로 Claude Code ROI 낮음 |

### "빠르게 실패하고 빠르게 피봇" 전략

| 일차 | 작업 | 검증 게이트 |
|------|------|-----------|
| 1-2일 | 현재 홈페이지를 랜딩페이지로 전환, 사전등록 수집 | - |
| 3-5일 | 역방향 검색 구현 (기존 6개 대학 활용) | 작동 확인 |
| 6-7일 | 과목 표기 정규화 | 검색 정확도 100% |
| 8-10일 | 입시 커뮤니티 배포 + 피드백 수집 | **Gate 1: 사전등록 50건?** |
| 11-14일 | 피드백 반영 또는 피봇 결정 | **Gate 2: WAU 100명?** |

### 핵심 메시지

> **"속도는 이 프로젝트의 근본 문제를 해결하지 못한다. Claude Code의 가속 효과는 '코드 중심 프로젝트'에서 극대화된다. 입시 데이터 프로젝트의 병목은 코드가 아니므로, Claude Code 리소스의 ROI가 낮다."**

## 3. 현실주의 비판자: "잘못된 방향으로 더 빨리 달리면, 목적지에서 더 멀어진다"

### "빠르면 성공한다"의 해부

6개 실패 원인 중 속도가 해결할 수 있는 것:

| 실패 원인 | 속도로 해결? | 증거 |
|-----------|------------|------|
| 사용자 검증 0회 | **불가능** | 6라운드 전체에 사용자 인터뷰 없음 |
| 데이터 출처 위조 | **불가능** | `course-data.ts:2` "ipsihogu.com" vs `page.tsx:111` "공식 모집요강" |
| 2028 데이터 95% 무효화 | **불가능** | 2022 개정 교육과정 전환 |
| 경쟁자 존재 | **불가능** | ipsihogu.com이 이미 운영 중 |
| 데이터 하드코딩 구조 | **가능** | 구조 개선 |
| 기능 부족 | **가능** | 역방향 검색 추가 |

**6개 중 2개만 속도로 해결 가능**

### "사람을 늘리면 된다"의 수학적 반박

```
1명 기대값: 0.25 x 500만원 - 0.75 x 200만원 = -25만원
2명 기대값: 0.30 x 500만원 - 0.70 x 400만원 = -130만원

→ 인력 추가 시 기대값 -25만원에서 -130만원으로 악화
→ "비용 증가 속도 > 성공 확률 증가 속도"
```

### 두 명이 되면 생기는 새로운 문제

1. **데이터 이원화 심화** — `course-data.ts`, `raw_recommendations.json`, `universities/*.json` 3곳의 데이터가 이미 불일치. 2인이 동시 수정하면 폭발
2. **의사결정 분산** — ipsihogu.com 데이터 계속 쓸 것인지, 역방향 검색이 핵심인지 등 방향성 결정이 지연
3. **책임 희석** — `naver_cafe_crawler.py`의 `COOKIE = ""`가 증명: 코드는 만들었지만 실행은 안 함. 병목은 코딩 능력이 아니라 실행 의지

### 진짜 질문은 무엇인가?

| 잘못된 질문 | 진짜 질문 |
|-----------|----------|
| "두 명이면 빠르지 않을까?" | **"학부모 10명에게 보여줬을 때 3명이 즐겨찾기에 등록하는가?"** |
| "Claude Code로 더 빨리 만들 수 있지 않을까?" | **"ipsihogu.com과 뭐가 다른가를 30초 안에 설명할 수 있는가?"** |
| "빠르게 완성하면 국책사업도 가능하지 않을까?" | **"2028 교육과정 전환 시 데이터를 어떻게 확보할 것인가?"** |

### 유일한 게임 체인저: 국책사업

| 국책사업 | 효과 |
|----------|------|
| 예비창업패키지 (5,000만원) | 기회비용 제거 + 데이터 인력 고용 가능 |
| 데이터바우처 (4,500만원) | PDF→JSON 가공 비용 전액 지원 |
| KERIS 소프트랩 | 공교육 데이터 접근 + 검증 |

국책사업이 기대값을 뒤집을 수 있지만, 국책사업에 필요한 것은 "속도"가 아니라 **"시장 검증 근거 + 데이터 합법성 + 차별화"**

### 핵심 메시지

> **"지금 필요한 것은 두 번째 개발자가 아니라, 첫 번째 사용자다."**

## 5. 4명의 최종 합의/불일치

### 전원 합의 (4/4)

| # | 합의 |
|---|------|
| 1 | **Claude Code는 코딩을 3-5배 가속하지만, 데이터 수집/검증은 가속할 수 없다** |

|------|----------------------|---------------------------|
| **"2인이면 기대값이 올라가는가?"** | | |
| 파워유저 | "코딩은 빨라지지만, 데이터 병목은 그대로. 4주가 현실적" | |
| 비판자 | **"기대값 -25만원→-130만원으로 악화. 비용 증가 > 확률 증가"** | |
| 속도 전문가 | | "단독 속도론 기대값 음수. 하지만 국책사업 변수 포함 시 전혀 다른 게임" |
| 팀 전략가 | | **"데이터 전담자 분리가 1인의 가장 큰 실패 원인을 해소. 기대값 +500~1,800만원"** |

### 핵심 분기점: "국책사업 없이 2인" vs "2인 + 국책사업"

```
[국책사업 없이 2인만]
  → 비용 2배, 성공 확률 +5%p
  → 기대값 악화 (-25만→-130만원)
  → 결론: 합리적이지 않음

[2인 + 국책사업]
  → 데이터 병목 해소 (A 전담)
  → 코딩 시간 74% 절약 (Claude Code)
  → 비대칭 수익 구조 (실패해도 서류 5시간 손실, 성공하면 수천만원)
  → 기대값 +500~1,800만원
  → 결론: 게임 체인저
```

### 질문에 대한 정직한 답

| 그러면 의미 없나? | **"2인+CC" 단독은 의미 약함** | 기대값이 오히려 악화 |
| **진짜 게임 체인저는?** | **"2인+CC+국책사업" 조합** | 기대값 -68.5만원 → +500~1,800만원 |

### 최종 메시지

> **7라운드의 가장 중요한 발견:**
>
> **"두 명의 개발자 + Claude Code"는 질문의 절반만 맞습니다.**
>
> 코딩 속도는 3-5배 빨라지지만, 이 프로젝트의 실패 원인 중 속도로 해결되는 것은 10%뿐입니다. "빠르게 만드는 것"이 아니라 "빠르게 검증하는 것"이 핵심입니다.
>
> 그러나 **국책사업이라는 변수를 포함하면 게임 자체가 바뀝니다.** 실패해도 잃는 것은 서류 작성 5시간이고, 성공하면 수천만원의 자금 + 공교육 실증 기회 + 데이터 가공 비용 지원을 얻습니다. 이 비대칭 구조가 기대값을 -68.5만원에서 +500~1,800만원으로 뒤집습니다.
>
> **결론:**
> 1. 먼저 **학부모 5명에게 물어보세요** (2시간)
> 2. 반응이 있으면 **2인+Claude Code로 2주 스프린트** (배포까지)
> 3. 배포 후 **국책사업 사업계획서 작성** (에듀테크 소프트랩 + 예비창업패키지)
> 4. 하이브리드로 **개발자 도구 비교 사이트 병행** (리스크 분산)
>
> **"두 번째 개발자"보다 "첫 번째 사용자"가 먼저입니다. 그리고 둘 다 있으면, 국책사업이 모든 것을 바꿉니다.**
