## 1. Pragmatic Developer (실전 개발자): "코드가 아니라 데이터가 병목"

### 현재 코드베이스 실태

프로젝트는 **소스 파일 5개, 약 620줄**의 초기 프로토타입입니다. 전체 앱이 `"use client"` CSR로 동작하여 SEO가 불가능하며, `generateStaticParams`가 전혀 구현되어 있지 않습니다.

| 파일 | 상태 | 문제 |
|------|------|------|
| `course-data.ts` (197줄) | 6개 대학 하드코딩 | JSON 미분리, ipsihogu.com 출처 |
| `guide/page.tsx` (373줄) | `"use client"` CSR | SSG 전환 미구현 |
| `next.config.ts` | 설정 비어있음 | SSG/output/이미지 최적화 없음 |
| `data/universities/*.json` (15파일) | 껍데기 | `required_courses`가 있는 것은 4개뿐 |
| `raw_recommendations.json` (725줄) | 데이터 이중화 | course-data.ts와 중복, 잘린 데이터 다수 |

### 기술 스택 평가

| 항목 | 판정 | 근거 |
|------|------|------|
| Next.js 16 | 적정 (과도하지만 전환 비용 > 이점) | SSG만 필요하지만, 이미 셋업됨 |
| React 19 | 적정 | Server Components는 SSG 전환 시 활용 |
| Tailwind 4 | 주의 | v4 CSS-first 방식은 자료 부족 |
| SSG 전략 | **절대적으로 필요** | 데이터 연 1회 업데이트, SEO가 트래픽 핵심 |

### 스프린트 계획의 현실적 시간 재산정

| 주차 | 계획 시간 | 현실적 시간 | 초과율 | 핵심 병목 |
|------|----------|-----------|--------|---------|
| Week 1-2 | 16h | **24h** | +50% | 3개 대학 추가 = 코딩이 아닌 데이터 수집 노동 |
| Week 3 | 10h | 8-10h | 0% | 역인덱스는 데이터 구조가 단순 |

### 숨겨진 기술적 함정 4가지

**1. 데이터 이중화 (Critical):** `course-data.ts`, `raw_recommendations.json`, `data/universities/*.json` — 세 데이터 소스의 관계가 정의되지 않음. Single source of truth 없음.

**2. 과목명 표기 불일치 (High):**
- 서울대 `"확률과 통계"` (띄어쓰기) vs 연세대 `"확률과통계"` (붙여쓰기) — 같은 파일 내
- `"화학I"` (영문 I) vs `"화학Ⅰ"` (로마자) — 교차 검색 시 누락 발생
- 역방향 검색에서 이 불일치는 **검색 누락**을 직접 유발

**3. 2028 vs 2026 데이터 모순:** 모든 데이터 파일의 `year` 필드가 `2026`이지만 프로젝트 목표는 `2028 대입 개편`. 2028학년도 데이터는 대부분 미발표 상태일 가능성.

**4. `compareUniversities` 매칭 로직 취약:**
```typescript
d.includes(department) || department.includes(d.replace(/학과|학부|전공|계열/g, ""))
```
"공학" 검색 시 기계공학부·화학공학과·건축공학과 모두 매칭되는 **오탐 문제**.

### Claude AI 10시간 절약은 현실적인가?

**YES — 코딩 영역에서는 8-13시간 절약 가능.** 코드베이스가 작고(620줄), 아키텍처가 단순하므로 JSON 분리, SSG 보일러플레이트, 역인덱스, OG 이미지, 컴포넌트 분리에서 효과적.

**NO — 데이터 수집/검증 영역에서는 AI가 도울 수 없음.** PDF 읽기, 2028 데이터 존재 여부 확인, 과목명 표기 통일 기준 결정은 수작업.

### 핵심 결론

> **5라운드 분석에서 간과된 핵심: "코드가 아니라 데이터가 병목." 코딩에 걸리는 시간보다 데이터 수집/검증/정규화에 걸리는 시간이 2-3배 클 것이다.**

## 2. Data Skeptic (데이터 회의론자): "30개 매핑은 빙산의 일각"

### "권장과목"의 의미론적 모호성

대학마다 "핵심권장과목"과 "권장과목"의 구속력이 다릅니다.

| 대학 | 표현 | 서비스에서의 매핑 | 실제 구속력 |
|------|------|----------------|-----------|
| 서울대 | "핵심권장과목" / "권장과목" 2단계 | `core` / `recommended` | **불명확** |
| 경희대 | "핵심" / "권장" | `core` / `recommended` | 서울대와 다를 수 있음 |
| 대교협 | "참고자료 (필수 이수 기준 아님)" | - | 전혀 다른 맥락 |

**동일한 `core`/`recommended` 필드가 대학마다 완전히 다른 구속력을 의미합니다.** 학부모가 "핵심과목"을 "안 들으면 불합격"으로 해석할 위험이 현실적으로 존재하며, 면책조항만으로 해결되지 않습니다.

### 정규화의 실제 복잡성: "30+ 매핑"은 심각한 과소평가

**현재 코드에서 이미 발견된 불일치:**

| 유형 | 예시 | 현재 처리 | 결과 |
|------|------|----------|------|
| 공백 차이 | "확률과 통계" vs "확률과통계" | 미처리 | **다른 과목으로 인식** |
| 로마자/영문 | "화학I" vs "화학Ⅰ" | 미처리 | **다른 과목으로 인식** |
| 데이터 잘림 | "화학Ⅱ, 미적분," (trailing comma) | 미처리 | **누락 데이터** |
| 자연어 조건 | "물리학Ⅱ 또는 화학Ⅱ 또는" | 하나의 문자열로 처리 | **파싱 불가** |
| 공백 문자열 | `" "` vs `""` | `filter(Boolean)` 미처리 | **빈 값 노출** |

`raw_recommendations.json`에서 확인된 잘린 데이터만 **최소 5건** (296행, 361행, 401행, 409행, 421행).

**30개 매핑으로 해결되는 것은 공백/로마자 차이뿐.** "또는" 조건문 처리, 잘린 데이터 복구, 학과명 크로스 매핑은 **데이터 모델 자체의 재설계**가 필요합니다. 현재 `DepartmentCourses` 인터페이스의 `core: string; recommended: string`은 조건부 권장을 표현할 수 없는 구조입니다.

## 3. Self-Deception Detector (자기기만 탐지기): "20명이 모두 Go라는 것 자체가 의심스럽다"

### 구조적 편향 분석

**AI의 "긍정적 결론 편향":** AI 언어모델은 사용자의 의도를 돕는 방향으로 학습됩니다. 사용자가 프로젝트 분석을 의뢰했다는 것 자체가 "하고 싶다"는 암묵적 신호. 20명 중 **단 한 명도 "이 분석 자체가 시간 낭비이니 멈추세요"라고 말하지 않았습니다.**

**역할 배정의 비대칭:**

| 라운드 | Go 방향 역할 | No-Go 방향 역할 |
|--------|-------------|----------------|
| 1라운드 | PM, Researcher, Architect (3) | Critic (1) |
| 2라운드 | Builder, Strategist (2) | Skeptic (1) |
| 3라운드 | Tech Lead, Data Strategist, Growth Hacker (3) | 0 |
| 4라운드 | Blue Team, Market Analyst (2) | Red Team (1) |
| 5라운드 | CEO, Execution Lead, Mentor (3) | Devil's Advocate (1) |

매 라운드 Go 방향이 2-3배 많고, 3라운드는 반대 역할이 **아예 없습니다.** 5라운드 "최종 의사결정자"를 CEO가 아니라 "청산인"이라 불렀다면 결론이 달라졌을 것입니다.

**라운드 누적 합의 압력:** 3라운드부터 "어떻게 할 것인가"를 논의하면서 "할 것인가 말 것인가"가 테이블에서 사라졌습니다. 4라운드 Red Team이 성공 확률을 15-20%로 낮춰도 이미 "실행 계획"이라는 프레임 안에 갇혀 있었습니다.

### "실패 비용 0원" 논증의 해체

**기회비용의 체계적 무시:**
- 심리적 전환비용: 실패 선언 후 다음 프로젝트까지 평균 3-6개월 간격
- 인지 자원 고갈: 출퇴근길, 잠들기 전 이 프로젝트를 생각하는 배경 인지 부하 → 본업 성과 영향
- **귀류법:** "실패 비용 0원"이 유효하면 성공 확률 0.1%인 프로젝트도 해야 함. 불합리.

**Devil's Advocate 자신의 모순:** 기대값 -68.5만원을 계산하고, 대안 기대값 +221.9만원을 제시하고도 Go를 결론. "더 좋은 아이디어를 찾다가 아무것도 안 하는 것이 가장 나쁘다"는 논리는 **"더 좋은 아이디어가 이미 있는 경우"를 배제**합니다.

### "학습 투자" 프레이밍 = 골대 이동

| 라운드 | 성공 기준 | 성공 확률 |
|--------|----------|----------|
| 1라운드 | 수익화 가능한 서비스 | (미제시) |
| 2라운드 | 월 100만+ 부수입 | 35-40% |
| 4라운드 | 4월 말 MVP | 20-30% |
| 5라운드 | **"학습 투자"** | **사실상 100%** |

성공 확률이 낮아질 때마다 성공의 정의가 완화. "학습"이 목표면 실패란 존재할 수 없고, **실패를 인식할 수 없으면 패턴 수정도 불가능.**

### Kill Switch는 작동하지 않는다

| 시나리오 | 예상 반응 | Kill Switch 작동? |
|----------|----------|-----------------|
| 3/28에 80% 완성 | "며칠만 더 하면 되는데" | **No** |
| 3/28에 30% 완성 | "데이터 작업이 예상보다 오래 걸렸을 뿐" | **No** |
| 3/28에 시작도 못 함 | 자연스럽게 포기 | Yes (하지만 필요 없음) |

**Kill Switch가 진짜 필요한 상황에서는 작동하지 않고, 작동하는 상황에서는 필요하지 않습니다.**

Kill Switch를 실효적으로 만들려면: (1) 제3자에게 공개 선언, (2) 중단 시 대안 프로젝트 사전 작성, (3) "중단은 의사결정 능력의 증거" 내면화.

### 진짜 정직한 결론

> **"이 프로젝트의 기대값은 마이너스이고, 사용자 검증은 0회이며, 데이터 출처에 법적 문제가 있고, 핵심 차별화 기능(역방향 검색)은 검증되지 않은 가설입니다. 그럼에도 하겠다면, 그것은 '합리적 의사결정'이 아니라 '하고 싶으니까 하는 것'입니다. 그리고 그것도 괜찮습니다 — 다만, '분석 결과 합리적이므로 Go'라고 포장하지는 마십시오."**

**EntranceExamPlanner를 현재 상태 그대로 즉시 배포하고, 남은 시간의 70%를 개발자 도구 비교(B-1)에 투자.**

| 주차 | EntranceExamPlanner | 개발자 도구 비교 |
|------|--------------------|--------------------|
| Week 1-2 | 현재 상태로 Vercel 배포 (추가 기능 없이) | 리포 생성, 비교 UI 복사, 5개 카테고리 선정 |
| Week 3-4 | Search Console 등록만 | 5개 카테고리 완성, 배포 |
| Week 5-8 | 트래픽 모니터링만 | 확장 + 어필리에이트 설정 |

### 의사결정 플로우차트

```
[이번 주] EntranceExamPlanner 현재 상태로 Vercel 배포
     |
[다음 주] 공유 인프라 구축 + 학교알리미 데이터 수집 가능성 조사
     |
     +-- 학교알리미 수집 가능? --YES--> [A-3] 학교별 매칭 확장 (같은 도메인)
     |

                                 +--> [B-4] AI 도구 비교 (한국어)
```

### 피봇 전략가의 핵심 메시지

> **"추가 기능에 57시간을 쏟는 것보다, 그 시간의 70%를 성공 확률이 더 높은 프로젝트에 투자하는 것이 기대값을 극대화한다. 가장 위험한 것은 프로젝트의 실패가 아니라, 기대값이 마이너스인 프로젝트에 계속 시간을 투자하는 것이다."**

## 5. 4명의 에이전트 최종 합의/불일치

### 전원 합의

| 1 | **데이터 품질이 5라운드에서 과소평가됨** | 잘린 데이터, "또는" 처리, 표기 불일치가 이미 존재 |

### 의견이 갈리는 사항

| 쟁점 | Pragmatic Developer | Data Skeptic | Self-Deception Detector | Pivot Strategist |
|------|--------------------|--------------|-----------------------|-----------------|
| 프로젝트를 계속 해야 하나? | 조건부 Yes (데이터 병목 해결 시) | 조건부 Yes (데이터 모델 재설계 시) | **"하고 싶으면 해라, 합리적이라 포장하지 마라"** | **No — 피봇이 합리적** |
| Kill Switch | 데이터 완성 여부로 기준 변경 | 데이터 품질 검증으로 기준 추가 | 외부 강제력 없이는 무의미 | 불필요 (피봇하면 됨) |
| 역방향 검색의 가치 | 기술적으로 쉬움, 가치 불확실 | 정규화 없이는 오류 투성이 | 검증되지 않은 가설 | 같은 기술로 더 큰 시장 가능 |
| 2022 개정 교육과정 | `output: "export"` 한 줄로 SSG 시작 | 2년 후 전면 재구축 필수 | 구조적 리스크 수용하며 Go | 수명 2년 제한 → 피봇 이유 |
