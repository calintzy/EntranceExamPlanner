## 1. Tech Lead: 역방향 검색 구현 설계

### 근본 원인 진단

> 모든 기술적 문제의 근본 원인은 **데이터 모델링 부재**. 과목/학과/대학이 고유 ID 없이 한글 문자열로만 존재하여, 동일 실체의 다른 표현("확률과 통계" vs "확률과통계")을 구분/통합 불가.

### 핵심 아키텍처: 3개 신규 데이터 구조

**1) 과목 정규화 사전 (`data/subjects/subject-dictionary.json`)**

```json
{
  "prob_stat": {
    "id": "prob_stat",
    "canonical": "확률과 통계",
    "category": "수학",
    "aliases": ["확률과통계", "확률과 통계", "확률통계"]
  },
  "chemistry2": {
    "id": "chemistry2",
    "canonical": "화학Ⅱ",
    "aliases": ["화학II", "화학2", "화학Ⅱ"]
  }
}
```

**2) 학과별 권장과목 JSON (`data/recommendations/seoul_nat.json`)**
- 과목은 ID로 참조 (문자열 아님)
- 빌드 타임에 subject-dictionary와 조인

**3) 학과 동의어 사전 (`data/meta/department-aliases.json`)**
- "컴퓨터공학부" = "컴퓨터과학과" = "소프트웨어학과" 클러스터링

### 역방향 검색 알고리즘

```
사용자 과목 선택 → SubjectId[] 변환
  → 역인덱스에서 해당 과목 요구하는 (대학, 학과) 수집
  → 정방향 데이터로 전체 요구 과목 확인
  → 가중 스코어링 → 정렬 → UI 렌더링

스코어링:
  총점 = (핵심권장과목 이수율 × 70) + (권장과목 이수율 × 30)
  등급: A(90+), B(70-89), C(50-69), D(0-49)
```

### SSG 전환 라우트 구조

```
/universities/[univId]/           대학별 허브 (SSG)
/universities/[univId]/[deptId]/  학과별 상세 (SSG) ★ SEO 핵심
/compare/                         비교 도구 (CSR)
/reverse/                         역방향 검색 (CSR)
```

- `generateStaticParams()`로 모든 대학×학과 조합 사전 빌드
- 15개 대학 확장 시 ~500-800 페이지, 빌드 수 초

### 구현 우선순위

```
1. 과목 정규화 사전 생성
2. recommendations JSON 분리
3. 빌드 타임 역인덱스 생성
4. 역방향 검색 UI
5. SSG 라우트 분리
6. 학과 동의어 사전
```

### 기술 스택 판정

- **유지**: Next.js 16 + React 19 + Tailwind 4 + JSON 파일
- **추가하지 말 것**: DB, CMS, ORM, API 서버, 상태관리 라이브러리
- **추가 권장**: `zod` (JSON 검증), URL 기반 상태관리 (SSG 호환)

## 2. Data Strategist: 데이터 수집 전략

### 현재 데이터 현황 (결정적 발견)

| 항목 | 현실 |
|------|------|
| 15개 대학 JSON | 존재하지만 **학과별 권장과목 데이터는 서울대만 구체적** |
| 나머지 14개 대학 | required_courses 필드 없거나 설명만 |
| 인문계열 | **데이터 0건** |
| 데이터 연도 | 2026학년도 (타겟은 2028) |

### 즉시 수집 가능한 가장 가치 있는 데이터

> **대교협이 2025년 9월에 이미 2028학년도 45개 대학 권장과목을 발표했고, 2026년 2월에 정리본까지 공개.** adiga.kr 자료실에서 지금 바로 다운로드 가능.

### 3단계 우선순위

| 순위 | 작업 | 소요시간 | 시기 |
|------|------|---------|------|
| 1순위 | 대교협 2028 권장과목 PDF → 15개 대학 추출·구조화 | 20-28시간 | 즉시 |
| 2순위 | 2028 시행계획 발표 → 전형정보 수집 | 대기 후 2주 | 2026년 4-5월 |
| 3순위 | 인문계열 + 예체능 데이터 보강 | 15-20시간 | 4-8주 |

### AI 자동화 파이프라인

```
PDF 다운로드 → pdfplumber 텍스트/표 추출 → Claude Haiku JSON 변환 → 자동 규칙 검증 → 수동 스팟체크
```

- **연간 API 비용: ~5,000원** (사실상 무시 가능)
- 병목은 비용이 아니라 PDF 품질과 프롬프트 튜닝

### 연간 유지보수 재추정 (Skeptic 75시간 비판 대응)

| 구간 | 주당 시간 |
|------|----------|
| 4-8월 (집중 수집기) | 주 3-4시간 |
| 9-3월 (유지기) | 주 0.5시간 미만 |
| **연 평균** | **주 0.9시간 (연 45시간)** |

- AI 파이프라인 없이 수동: 55-65시간/년
- AI 파이프라인 적용 시: **45시간/년**
- Skeptic의 75시간은 전체 학과 대상 + AI 미적용 전제에서 과대추정

## 3. Growth Hacker: SEO & 마케팅 전략

### 키워드 전략: 3티어 접근

| 티어 | 키워드 예시 | 전략 |
|------|-----------|------|
| 1티어 (고경쟁) | "대입 권장과목", "고교학점제 과목선택" | 장기 목표, 직접 공략 불가 |
| 2티어 (중경쟁) | "서울대 컴공 권장과목", "2028 대입 개편 권장과목" | **3-6개월 내 학과별 정적 페이지로 공략** |
| 3티어 (롱테일) | "미적분 안 들으면 갈 수 있는 대학" | **즉시 공략, 누적 효과** |

### 2028 대입 개편 키워드: 블루오션 발견

> `2028 대입 권장과목 비교`, `2028 고교학점제 과목선택` 등은 **아직 전용 서비스 페이지가 없는 신생 키워드**. 지금 진입하면 선점 가능.

### SEO 페이지 URL 체계

```
/university/[slug]           → 대학별 허브 (6개)
/university/[u]/[dept]       → 학과별 상세 (~170개) ★
/subject/[subject]           → 과목별 역방향 (신규)
/major/[majorSlug]           → 계열별 허브
/blog/5개-핵심-콘텐츠         → SEO 앵커
```

동일 데이터로 3개 축(대학/과목/계열) 인덱싱 가능.

### 네이버 vs 구글 이원화

| | 구글 | 네이버 |
|---|---|---|
| 전략 | SSG 정적 페이지 + sitemap | 별도 블로그 운영 필수 |
| 이유 | 외부 사이트 잘 크롤링 | 자체 서비스(블로그/카페) 우선 |
| 실행 | Search Console 등록 | 네이버 블로그 "고교학점제 연구소" 개설 |

### 바이럴 메커니즘

1. **과목 조합 체크 공유**: `/check?subjects=calculus,physics2` → 카카오톡 OG 이미지 자동 생성
2. **권장과목 카드 저장**: 인스타/카톡 공유용 세로형 이미지 + 워터마크
3. **역발상 랭킹**: "과목 부담 없는 대학 순위", "이공계 가장 까다로운 학과 TOP 10"

### "AI" 타이틀 즉시 수정 권고

```
현재: "입시플래너 - AI 교과 선택 가이드" (AI 기능 없음)
권장: "어떤 과목 들어야 할까? 대학별 권장과목 한눈에 비교"
```

### 1번 리스크 상세: ipsihogu.com 데이터 문제

**결정적 불일치 발견:**

| 위치 | 내용 |
|------|------|
| `course-data.ts` 2행 (코드) | `// 출처: ipsihogu.com` |
| `page.tsx` 111행 (사용자 UI) | `데이터 출처: 각 대학 입학처 공식 모집요강` |

**법적 문제:**
- **부정경쟁방지법 제2조 1호 차목**: 타인의 성과물 무단 사용 (영리 목적 불요)
- **저작권법 제93조**: 데이터베이스제작자 권리 침해 가능
- **출처 허위 표기**: 의도적 기만으로 해석될 위험

**즉시 대응:**
1. 모든 데이터를 대학 입학처/대교협에서 직접 재수집
2. ipsihogu.com 출처 주석 제거
3. 또는 ipsihogu.com에 데이터 활용 협의 요청

### "반드시 이수" 표현 수정 필요

```
현재 (guide/page.tsx 184행): "반드시 이수해야 하는 과목"
수정안: "대학이 전공 연계를 위해 특히 권장하는 과목"
```

근거: 대교협 자체가 "필수 이수 기준이 아닌 참고자료"라고 명시.

### 면책조항 설계 (3단계 배치)

1. **이용약관 페이지** (별도 신설) — 법적 면책 범위 명확화
2. **가이드 페이지 상단** — 결과 조회 전 주의사항 노출
3. **푸터** — 간결한 참고용 안내 + 데이터 기준일

### 대학 모집요강 데이터 자체의 저작권

> 대학 모집요강은 **공공기관의 고시/공고에 해당하여 저작권법 제7조 제4호에 따라 보호 대상이 아님.** 개별 사실 데이터도 저작물이 아님. → 직접 수집하면 법적 문제 없음.

### 개인정보보호법 (미성년자 주의)

- 현재: 개인정보 수집 없음 → 별도 조치 불필요
- 수익화 시: 만 14세 미만 법정대리인 동의 필요 (고등학생은 만 15-18세이므로 직접 대상 아니나, 예비고1 중3 주의)

## 6. 3라운드 핵심 결론

### 가장 긴급한 3가지 (서비스 공개 전 필수)

1. **ipsihogu.com 데이터 교체** — 대교협/입학처 직접 수집으로 전환 (법적 시한폭탄)
2. **"AI" 타이틀 제거** — 코드 1줄 변경으로 즉시 해결
3. **"반드시 이수" 문구 수정** — 대교협 공식 입장과 불일치

### 가장 가치 있는 즉시 행동

> **adiga.kr에서 "2028학년도 권역별/대학별 권장과목" PDF를 지금 바로 다운로드하라.** 이것이 이 서비스의 핵심 가치를 실현할 수 있는, 현재 시점에서 확보 가능한 유일한 2028 데이터.

### 기술적 핵심 전환

```
현재: 문자열 기반 하드코딩 → 매칭 실패, 확장 불가, SEO 불가
목표: ID 기반 정규화 + JSON 분리 + 빌드 타임 역인덱스 + SSG
```

이 전환이 완료되면 역방향 검색, SSG, 횡단 비교가 모두 동작하는 구조가 됩니다.
